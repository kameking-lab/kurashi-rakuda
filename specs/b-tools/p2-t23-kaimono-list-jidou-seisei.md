# 買い物リスト自動生成（献立連動）（P2-T23）

## 概要

献立自動提案（Q3-16 `kondate-teian`）で使っているレシピ・食材データ（`data/kondate`）をそのまま流用し、
利用者が選んだ複数のレシピに必要な食材をまとめて「買い物リスト」として表示するツール。

- docs/02 **#44**（カテゴリ3: 家事・料理、需要 中、難易度A、YMYL 低。**#40（献立自動提案）の拡張**）。BACKLOG.md queueId: `P2-T23`。
- 依存データ: `data/kondate/recipes.json`・`data/kondate/ingredients.json`（**Q3-16 部品流用**。新しいデータファイルは作らない）。
  - `lib/tools/impl/kondate-teian.ts` の `kondateData`・`ingredientMap`・`resolveIngredientId` をそのまま呼び出す。
  - レシピの抽選（乱数）ロジックは使わない。本ツールは「利用者が選んだレシピ」の食材を集計するだけの決定的な集計ツールであり、抽選要素はない。
- YMYL区分: 低（金額・制度・医療のいずれにも関わらない生活の目安ツール）。

## 前提となるデータ制約（★実装前に必ず確認すること★）

`data/kondate/recipes.json` の各レシピの `ingredients` 配列は `{ id, role }`（食材id・役割）のみを持ち、
**分量（数量・単位。例: "200g"・"大さじ2"）を一切持たない**。`data/kondate/ingredients.json`（食材マスタ）にも
分量に相当するフィールドはない。

これは献立自動提案（Q3-16）の設計思想（specs/s-tools/06-kondate-teian.md）が「一般的な家庭料理の名称・主材料・
調理法だけを持ち、分量や手順文は持たない」ことを明言しているためであり、データの欠落や取得漏れではない。

存在しない分量データを本ツール側で仮定して生成する（例:「レシピは2人分が基準だから、肉は1人前100gとみなす」）ことは、
根拠のない数値を「買い物量」として提示してしまうリスクがある。本サイトの一貫した設計方針（わからないことは
わからないと言う。docs/09 らっくの口調・kondate-teian の `excludeNotice`/`reuseSentence` と同じ姿勢）に従い、
本ツールは**分量の合算は行わない**。代わりに「同じ食材を何品のレシピで使うか（使用回数）」を集計し、
「実際の分量は各レシピを参照してください」と明示する。

## 集計の単位（★合算ルール★）

- 集計のキーは **食材id + 役割（role）** の組（例: `onion` かつ `role=main`）とする。
- 理由: `data/kondate/recipes.json` には同じ食材idでも役割（`main`/`sub`/`garnish`/`seasoning`）が
  レシピによって異なるケースが実在する（例: 玉ねぎ〈`onion`〉は主材料として使われるレシピと、
  付け合わせ・副材料として使われるレシピの両方がある。実データで27/93食材が複数roleで登場する）。
  役割が違えば実質的に必要な分量感・買い方も異なるため、**役割が異なるものを1行に無理に合算しない**
  （課題文の「単位が異なる材料の合算は無理に変換せず、単位ごとに分けて表示する」に対応する、
  本データにおける最も正直な解釈である。物理単位〈g/大さじ等〉のデータが存在しないため、
  役割〈role〉を「実質的な単位区分」として扱う）。
- 同じ食材id・同じ役割であれば、選択したレシピをまたいで1行に集約し、「使用回数（何品のレシピで使うか）」を
  カウントする。同じレシピを重複して選択した場合は1回分としてのみ数える（レシピidの集合として正規化する）。
- 各行には「どのレシピで使うか」（レシピ名の一覧）を必ず添える。分量が分からない分、
  利用者が元のレシピを参照して分量を確認できるようにするための代替情報である。

## 入力仕様

| フィールド名 | 型 | 必須/任意 | バリデーションルール |
|---|---|---|---|
| 選択レシピid一覧（recipeIds） | 文字列配列 | 任意（0件可） | 0件のときは空の買い物リストを返す（エラーにしない。エッジケース参照）。`data/kondate/recipes.json` に存在しないidは無視し、`unknownRecipeIds` に集めて呼び出し側に伝える（エラーにはしない） |

UIはチェックボックスによるレシピ選択とする（`data/kondate/recipes.json` の全76レシピを主菜・副菜・汁物の3群に分けて表示。
コースごとの見出しで分類し、レシピ名で検索する簡易フィルタも提供する）。選択状態はURLクエリ（`r`パラメータ、
レシピidをカンマ区切り・id昇順で正規化）に保存し、URLの共有で同じ選択を再現できるようにする
（`kondate-teian` の `encodeConditions`/`decodeConditions` と同じ設計思想）。

献立自動提案（`kondate-teian`）が提案した1週間分のレシピをそのまま引き継ぐ機能は、両ツールの状態を
仲介する共有ストレージ・URL連携の追加実装が必要になり本チケットの「簡易仕様」の範囲を超えるため、
本バージョンでは実装しない（本ツール単体でレシピを選んで買い物リストを作る独立ツールとする）。
将来的に連携する場合の入口として、`buildShoppingList` はレシピidの配列を受け取るだけの純関数にしてあり、
`kondate-teian` の `WeekMenu.days` からレシピidを取り出して渡せば動く設計にしている。

## ロジック仕様

### 1. 集計処理（`buildShoppingList`）

```
入力: recipeIds（文字列配列）, data（KondateData、既定は kondateData）
1. recipeIds を重複排除する（Set化）
2. 各idについて data.recipes から検索する
   - 見つかった → usedRecipes に加える
   - 見つからない → unknownRecipeIds に加える（エラーにしない）
3. usedRecipes を id 昇順に正規化する（入力の順序に依存させない。kondate-teian の候補プールと同じ方針）
4. usedRecipes の各レシピの ingredients を走査し、(ingredientId, role) をキーに
   「このキーを必要とするレシピidの集合」を作る（Setなので同じレシピの重複列挙があっても1回に潰れる）
5. キーごとに1行（ShoppingListItem）を作る
   - name: ingredients.json の name（見つからなければ id をそのまま表示）
   - roleLabel: role の日本語ラベル（主材料/副材料/薬味・付け合わせ/調味料）
   - count: そのキーを必要とするレシピ数
   - recipeNames: そのキーを必要とするレシピ名の一覧（レシピid昇順→名前に変換）
   - category / shelf / commonness: ingredients.json のマスタ値をそのまま転記（未登録食材は category="other", shelf="room", commonness=1 をフォールバックとして使い、クラッシュさせない）
6. items を 保管場所（shelf: room→fridge→freezer）→ 食材id → role の順に安定ソートする
   （店内の売り場に近い順で買い物しやすくするため。買い物リストとして提示する以上、
   並び順が実行のたびに変わらないことが利用者にとって重要）
```

### 2. 表示用ラベル

| role | roleLabel |
|---|---|
| main | 主材料 |
| sub | 副材料 |
| garnish | 薬味・付け合わせ |
| seasoning | 調味料 |

| shelf | 表示名 |
|---|---|
| room | 常温 |
| fridge | 冷蔵 |
| freezer | 冷凍 |

### 3. 常備品への配慮（保証ではなく事実の提示）

`ingredients.json` の `commonness`（1〜5。5=常備前提。塩・しょうゆ等）が4以上の食材には、
「常備していることが多い食材です（買い足しが不要な場合があります）」という注記を添える。
★これは「買わなくてよい」という断定ではない★（kondate-teian の `excludeNotice` が
「卵不使用です」ではなく「卵を含むレシピを除いています」と動作の説明に留めているのと同じ姿勢で、
「一般に常備されがちだ」という事実の提示に留め、リストから除外はしない）。

## エッジケース

- **レシピを1件も選んでいない**: エラーにせず、`items: []` の空リストを返す。UI側は
  「レシピを選ぶと、必要な食材がまとまって表示されます」という案内を表示する（未入力＝異常ではない）。
- **存在しないレシピidが渡された**: 該当idを無視し `unknownRecipeIds` に入れて処理を継続する
  （クラッシュさせない。他の正しいidの集計結果には影響しない）。
- **同じレシピidを複数回指定**: 重複を1件として扱う（Setで正規化するため、集計結果は1回選んだ場合と同じになる）。
- **同じ食材が同じレシピ内の ingredients に複数回登場する（データ上想定しにくいが防御的に）**: レシピidの
  Setで管理しているため、同じレシピからの重複は count に影響しない（1レシピにつき最大1カウント）。
- **同じ食材idが役割違いで複数のレシピにまたがって使われる**: 役割ごとに別行として表示する（上記「集計の単位」参照）。
- **全76レシピを選択した場合**: 集計・ソートが破綻しないこと（負荷・パフォーマンス面でも一覧表示できること）。
- **食材マスタに存在しないid（データ破損）**: 表示名をidそのままにフォールバックし、category/shelf/commonnessも
  安全な既定値にフォールバックする（クラッシュさせない）。
- **URLの `r` パラメータに不正な文字列・空文字が混じる**: デコード時にidとして扱えない要素は無視する
  （壊れたURLで真っ白な画面にしない）。

## YMYL配慮事項

- 本ツールは金額・医療・制度のいずれも扱わない、生活の目安（買い物リストの整理）専用のツールである（YMYL区分: 低）。
- 分量（数量・単位）を一切表示しない。存在しないデータを仮定して表示することは、実際の買い物量を誤らせる
  リスクがあるため行わない。この制約は画面に常時明示する（結果の直上に固定表示するCallout）。
- 「常備品かもしれません」という注記は、あくまで一般的な常備度に基づく参考情報であり、各家庭の在庫を
  保証・断定するものではないことを明示する。
- アレルギー対応・栄養価判断は行わない（kondate-teian と同じ、本サイト全体の一貫した方針）。

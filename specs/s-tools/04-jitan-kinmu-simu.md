## 1. 概要

### 1.1 何を解くツールか

育児のために所定労働時間を短縮して働く（＝時短勤務する）人が、**「時短にしたら毎月いくらになるのか」を1画面で把握できない**という問題を解く。

実際の手取りは次の4つが同時に動くため、単発の給料計算では答えが出ない。

1. **賃金そのものが減る**（時短の比率に応じて）
2. **育児時短就業給付金が乗る**（2025年4月創設。賃金の10%。非課税）
3. **社会保険料はすぐには下がらない**（標準報酬月額は時短の翌月に自動で下がらない）
4. **2026年4月から子ども・子育て支援金（0.23%）が新設され、控除が増えた**

本ツールは、この4つを1つのタイムラインに載せて「時短直後の月」と「随時改定後の月」を並べて提示する。

## 2. スコープ

### 2.1 含むもの

- 育児時短就業給付金（雇用保険法**第61条の12**）の支給可否判定と月額の算定
- 時短後の賃金からの社会保険料（健康保険・厚生年金）・雇用保険料・子ども・子育て支援金の概算控除
- 標準報酬月額が**すぐには下がらない**ことによる「時短直後」と「随時改定後」の手取り差の提示
- 支給限度額・最低限度額による頭打ち／不支給の警告
- 各月の支給要件4つのチェックリスト表示
- 「6時間勤務でないと対象外」という誤解の明示的な打ち消し

### 3.1 入力しない項目（意図的）

- **時短の程度が「1日6時間」かどうか** — 聞かない。要件ではないため（`noMinimumReduction`）。聞くと「6時間でないと駄目」という誤解を強化してしまう。
- **賞与** — 聞かない。開始時賃金月額の算定から賞与は除かれる（`data.ikujiKyugyoKyufuKin.wageDailyFormula.note`「臨時の賃金・3か月を超える期間ごとに支払われる賃金（賞与）は除く」）。

### 4.1 STEP 1: 有効期限チェック（★最初に実行する★）

```
if ( 計算実行日 > EXPIRES ) or ( jitanStartDate > EXPIRES ):
    → 計算を実行せず、§7.1 の期限切れ画面を表示して終了
```

- **根拠**: `data/seido/ikukyu-kyufu.json` の `amendments[0]`（`status: "expires"`, `expiresOn: "2026-07-31"`）／出典: 育児休業等給付の内容と支給申請手続（令和7年8月1日改訂版）／厚生労働省
- 同 `amendments[0].impact`: 「★2026年8月1日に必ず数値が変わる★ 2026-07-17時点で令和8年8月1日適用の新額は未公表（例年7月下旬公表）。本データは2026年7月31日まで有効。」
- また `supportLimit.amendmentNote` / `minimumAmount.amendmentNote` はいずれも「★令和8年7月31日までの額★」と明記している。

### 4.2 STEP 2: 制度要件の判定（対象になるか）

```
E1 = ( 子が2歳に達していない )
E2 = ( continuedFromIkukyu == true ) or ( insuredMonths12 == "yes" )
対象 = E1 and E2
```

- **E1 根拠**: `data.ikujiJitanShugyoKyufuKin.targetChildAge.value` = 「2歳に満たない子」／出典: 雇用保険法（e-Gov法令検索）
- **E2 根拠**: `data.ikujiJitanShugyoKyufuKin.eligibility.value`（「①2歳未満の子を養育するために1週間当たりの所定労働時間を短縮して就業する被保険者であること。②育児休業給付の対象となる育児休業から引き続き…または、育児時短就業開始日前2年間に賃金支払基礎日数が11日以上ある（ない場合は80時間以上ある）完全月が12か月あること。」）／出典: 育児時短就業給付の内容と支給申請手続／厚生労働省
- `insuredMonths12 == "unknown"` のときは**対象外と断定しない**。「ハローワークでの確認が必要です」と表示したうえで、参考値として計算を続行する（§7.3）。
- **時短の程度は判定に使わない**。根拠: `data.ikujiJitanShugyoKyufuKin.noMinimumReduction.value`「短縮後の1週間当たりの所定労働時間に上限・下限はない」、同 `.note`「『6時間勤務でないと対象外』は誤り」。

### 4.3 STEP 3: 各月の支給要件（4つ）

`data.ikujiJitanShugyoKyufuKin.monthlyEligibility.value` の4項目をそのまま判定に使う（出典: 育児時短就業給付の内容と支給申請手続／厚生労働省）。

| # | JSON の文言（`monthlyEligibility.value[n]`） | 入力 | 不成立なら |
|---|---|---|---|
| 0 | 初日から末日まで続けて被保険者である月 | `insuredWholeMonth` | その月は不支給 |
| 1 | 1週間当たりの所定労働時間を短縮して就業した期間がある月 | `weeklyHoursAfter < weeklyHoursBefore` または `inputMode=direct` かつ `wageMonthAfter < wageMonthBefore` | その月は不支給 |
| 2 | 初日から末日まで続けて育児休業給付又は介護休業給付を受給していない月 | `receivingOtherBenefit` | その月は不支給 |
| 3 | 高年齢雇用継続給付の受給対象となっていない月 | `koureishaKeizoku` | その月は不支給 |

★要件2の読み方の罠★ 原文は「**初日から末日まで続けて**育児休業給付又は介護休業給付を**受給していない**月」である。すなわち「月の一部で育休給付を受けた」場合の可否は原文からは一意に読み取れない。**この曖昧さを解消する情報は `data/seido` に無い**ため、`receivingOtherBenefit = true` のときは金額を出さず「復職月など育休給付と重なる月は、ハローワークにご確認ください」と表示する（§7.3）。

### 4.4 STEP 4: 支給額の算定（3区分）

根拠: `data.ikujiJitanShugyoKyufuKin.calculationRules`（`rules[0..2]`）／出典: 育児時短就業給付の内容と支給申請手続（令和7年8月1日時点版）／厚生労働省

```
W0 = wageMonthBefore   # 育児時短就業開始時賃金月額
Wm = wageMonthAfter    # 支給対象月に支払われた賃金額
T90 = W0 * 0.9

# --- 区分① : rules[0] ---
if Wm <= T90:
    A = floor( Wm * RATE )

# --- 区分② : rules[1] ---
elif T90 < Wm < W0:
    A = ★算式未確認★ → 数値を出さない（§7.3）

# --- 区分外（データに定めがない） ---
else:   # Wm >= W0
    A = 算定不能 → 「時短前より賃金が減っていないため、本ツールでは算定できません」

# --- 区分③（頭打ち） : rules[2] ---
if A + Wm > LIMIT:
    A = LIMIT - Wm

# --- 最低限度額 ---
if A <= MIN:
    A = 0（不支給）
```

#### 区分ごとの根拠の対応付け

| 区分 | JSON の `case` / `formula` | 本仕様の実装 |
|---|---|---|
| ① | `rules[0].case`: 「支給対象月の賃金額 ≦ 育児時短就業開始時賃金月額 × 90%」／`rules[0].formula`: 「支給額 = 支給対象月に支払われた賃金額 × 10%」 | `A = floor(Wm × RATE)` |
| ② | `rules[1].case`: 「育児時短就業開始時賃金月額 × 90% < 賃金額 < 同 × 100%」／`rules[1].formula`: 「支給額 = 支給対象月に支払われた賃金額 × 調整後の支給率」 | **数値を出さない**（下記） |
| ③ | `rules[2].case`: 「上記による支給額と賃金額の合計 > 支給限度額」／`rules[2].formula`: 「支給額 = 支給限度額 − 支給対象月に支払われた賃金額」 | `A = LIMIT − Wm` |

#### ★区分②を実装しない理由（最重要の設計判断）★

`rules[1].note` を全文引用する:

> ★未確認★ 調整後の支給率の厳密な算式は、パンフレットPDFの数式が段組みで分断されており原文レイアウトを再現できなかった。法文（雇用保険法第61条の12第6項第2号）は『育児時短就業開始時賃金日額に30を乗じて得た額に対する当該賃金の額の割合が100分の90を超える大きさの程度に応じ、100分の10から一定の割合で逓減するように厚生労働省令で定める率』と定めるのみ。実装時は雇用保険法施行規則の該当条文を必ず参照すること。趣旨は『賃金額と支給額の合計が育児時短就業開始時賃金月額を超えないよう支給率を調整する』こと。

したがって MVP では:

- **区分②では金額を一切表示しない。** 「10%を掛けて出す」暫定実装は**禁止**（過大表示になる。逓減後の率は必ず10%未満）。
- 代わりに「時短による減額が10%未満のため、給付の支給率が調整されます。金額は本ツールでは算定できません（ハローワークにご確認ください）」と表示し、**上限の目安として「賃金額と支給額の合計が時短前の賃金月額（W0）を超えることはありません」**（＝ `A < W0 − Wm`）という `rules[1].note` 記載の趣旨のみを提示する。
- 未収集項目としての追跡先: [queue/hoikuryo-backlog.md §8](../../queue/hoikuryo-backlog.md)「育児時短就業給付の『調整後の支給率』の算式 ／ 出典候補: 雇用保険法施行規則」。
- 施行規則の該当条文が収集され次第、`ikukyu-kyufu.json` に `rules[1].formula` の確定値を追加し、本節を差し替える。

#### 6-3. 計算式

```
# 標準報酬月額の代用（★等級表が未収集のため、報酬月額をそのまま用いる概算★）
SR_A = wageMonthBefore
SR_B = wageMonthAfter

社保(X)  = floor(SR_X * 0.0915) + floor(SR_X * HEALTH/2) + floor(SR_X * KODOMO/2)
雇用保険 = floor(wageMonthAfter * EMPINS)
税引前手取り(X) = wageMonthAfter - 社保(X) - 雇用保険
月の合計(X)     = 税引前手取り(X) + A        # A は非課税のためそのまま加算
```

- **`A` を非課税として加算する根拠**: `data.handoriJuwariSoutou.taxExempt.value = true`（`.label`「育児休業等給付の非課税」、`.note`「所得税・住民税ともに非課税。翌年の住民税・保育料の算定基礎にも入らない（＝復職後の保育料が下がる要因になる）」）／出典: 育児休業等給付の内容と支給申請手続／厚生労働省。育児時短就業給付は雇用保険法上の「育児休業等給付」に含まれる（`ikukyu-kyufu.json` の `sources[0].note`「第61条の6〜第61条の12。…の根拠条文」）。
- 「保育料の算定基礎に入らない」点は #21 保育料計算へのリンク導線として活用する（金額計算は本ツールでは行わない）。
- **所得税・住民税は計算しない**（§2.2）。「税引前」であることをラベルに必ず入れる。

### 9.2 ★浮動小数点の罠（最優先の実装注意）★

`RATE = 0.1` は2進浮動小数で正確に表現できない。TC-11（`Wm=24,110` → ちょうど `MIN=2,411`）と TC-12（`Wm=24,120` → `2,412`）は**1円の差で支給・不支給が分かれる**。

```js
// NG: 24110 * 0.1 = 2410.9999999999995 → floor で 2410 → 誤って「支給」判定
// OK: 整数演算にする
const A = Math.floor(Wm * 10000 * RATE / 10000);   // まだ危うい
const A = Math.floor((Wm * Math.round(RATE * 1000)) / 1000);  // RATE を千分率整数化して先に整数乗算
```

- 推奨: 制度データの `rate` を**千分率の整数**に変換してから乗算し、最後に除算する。
- 同様に `W0 * 0.9` の比較も `Wm * 10 <= W0 * 9` と整数比較に書き換えると安全（TC-03/TC-04 が保護される）。

### 9.3 「時間比例」は法令の算式ではない

`inputMode = byHours` の時間比例（`Wm = W0 × hoursAfter / hoursBefore`）は**本ツールの入力補助**であり、法令にも `data/seido` にも根拠はない。実際の賃金は基本給・手当の設計で決まる。UI に「時間比例で概算しています。実際の給与は勤務先の規程によります」と必ず表示し、`direct` 入力への切替導線を置くこと。

### 9.6 テスト設計上の注意（YMYL ダブルチェック）

[08_タスク分解](../../docs/08_タスク分解.md)「YMYL の金額計算（S/A）は必ず『実装者≠検算者』のダブルチェックを構造化する」に従い:

1. **実装者**は §4 の計算式のみを見て `packages/calc/jitan-kinmu/` を実装する。
2. **検算者**は §6 のテストケースの期待値を、`data/seido/ikukyu-kyufu.json` の一次データから**独立に**再計算して検証する（実装コードを読まずに）。
3. TC-02 は `calculationRules.example` に厚労省パンフレット由来の期待値（41,393円）が JSON 内に存在するため、**両者が一致することを最初に確認する**（錨として最も強い）。
4. 区分②（TC-04, TC-17）で金額が出ていないことを、検算者が明示的に確認する。**「なんとなく数字が出ている」ことを正常と見なさない。**

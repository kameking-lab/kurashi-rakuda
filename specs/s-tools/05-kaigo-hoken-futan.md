## 1. 概要

### 1.1 何を解くツールか

「親（または自分）が介護保険サービスを使ったら、**月にいくら払うのか**」を、次の 4 つの制度を組み合わせて求める。

| 制度 | 求めるもの | データのパス |
|---|---|---|
| ① 利用者負担割合 | 1割 / 2割 / 3割 | `data.futanWariai` |
| ② 区分支給限度基準額 | 要介護度ごとの上限単位数と、超過分の全額自己負担 | `data.kubunShikyuGendo` |
| ③ 高額介護サービス費 | 月の自己負担の上限額（超過分は払い戻し） | `data.kougakuKaigoServiceHi` |
| ④ 特定入所者介護サービス費（補足給付） | 施設入所時の食費・居住費の負担限度額 | `data.hojokyufu` |

### 1.2 競合との差別化（[02_ツール一覧](../../docs/02_ツール一覧.md) の記述の範囲で）

- 同 #90 の記述: 「**入力型シミュレーター不在の空白**。kaigo-hokenryou（保険料）とは別物＝重複なし」。優先ツール TOP10 では 9 位「シミュレーター不在の空白＋ペルソナ5開拓」。
- 同 §棲み分け方針: kurashi-keisan.jp の `kaigo-hokenryou` は**保険料**（払い込む側）。本ツールは**自己負担**（サービスを使った側）で、別制度。送客リストにも載っていない。
- 同 #93「高額介護サービス費 該当チェック」は本ツールの ③ を単独ツール化したもので Phase 2。本ツールは ③ を**内蔵**する（複合シミュレーションが本サイトの立ち位置）。
- 差別化の実体:
  1. **AND 条件を正しく実装すること**（§4 ステップ2）。負担割合の 2 要件を「または」で実装すると、1割の人を2割と誤る。
  2. **2026年8月1日の改正を境に数値を切り替えること**（§4 ステップ6）。`beforeAug2026` / `fromAug2026` の両方を持っている（`data.hojokyufu`）。
  3. **要介護度を状態像で説明しないこと**（§9.2-1）。まとめサイトが一律に書いている「要介護3＝全面的な介護が必要」等は**法令・告示に根拠がない**（`data.yokaigoNintei.noStateDefinition`）。書かない、という差別化。

※ 需要・競合の詳細は [02_ツール一覧](../../docs/02_ツール一覧.md) の記述以上のことを本仕様書に書かない。

### ステップ2: 負担割合の判定（★AND 条件★）


### ステップ4: 区分支給限度基準額と在宅サービスの自己負担

```

limitUnits = level.units
usedUnits  = 入力12 ?? limitUnits

withinUnits = min(usedUnits, limitUnits)
overUnits   = max(0, usedUnits - limitUnits)

自己負担 = round(withinUnits * tanka * 負担割合) + round(overUnits * tanka)   // 超過分は全額
```

## 9. 実装メモ（罠・端数処理・データ更新で壊れる箇所）

### 9.1 端数処理

- **単位数 × 単価**: `data.kubunShikyuGendo.levels[].yenAt10` が「単位数 × 10」ちょうどであることを、`units * 10 === yenAt10` で全 8 件アサートするテストを置く（データ改変の検知）。
- **単価が 10.0 以外の場合の端数**: 1〜7級地の単価は小数（例 11.4 円）。`10 × 1.14` の**丸め規則は本データセットに記載がない**（告示の規定は `data.tanka.description`「10円に、地域区分およびサービス種類に応じた割合を乗じた額」まで）。→ §7 の通り**1〜7級地では確定額を出さない**設計にすることで、根拠のない丸めを実装せずに済ませる。★これは仕様上の意図的な回避であり、あとで単価データが揃うまで丸めを実装しないこと。★
- **負担割合の乗算**: `36217 * 10 * 0.3 = 108651`（割り切れる）。浮動小数の 0.1/0.2/0.3 を使うと `0.1*167650 = 16765.000000000002` 等が出る。→ **整数演算に変換する**（`yen * rate10 / 10` where `rate10 ∈ {1,2,3}`）。
- **食費・居住費**: すべて円/日の整数。日数を掛けるだけ。丸め不要。
- **高額介護サービス費の払い戻し**: `max(0, 自己負担 - limit)`。端数処理なし。

### 9.2 罠（優先度順）

1. **★★状態像を書かない★★**（`data.yokaigoNintei.noStateDefinition`）。UI ラベル・ツールチップ・併走記事・OGP・meta description のどこにも「要介護3＝全面的な介護が必要」「要介護1＝部分的な介護」等を書かない。厚労省の一次資料は「その方の病気の重さと要介護度の高さとが必ずしも一致しない場合があります」と明示している。参考資料『要介護状態区分別の状態像』は**一次判定の中間評価項目の得点分布を示す統計的な参考図であり、認定基準ではない**（同 `note`）。→ T-38。**lint ルールで禁止語（「全面的な介護」「部分的な介護」「寝たきり」「身の回りのことができない」等）を機械検知する**（[08_タスク分解](../../docs/08_タスク分解.md) P1-05 の忠実性ゲートに載せる）。
2. **★負担割合の AND★**（`data.futanWariai.note`）。`||` で書くと、合計所得だけ高い人（例: 合計所得165万・年金収入のみ270万）を2割と誤る。実際、2割は利用者の約4.3%しかいない（`currentDistribution.value`）。→ T-05 / T-06 / T-09
3. **★非課税世帯・生活保護は課税所得では判定できない★**。`brackets[key=hikazei]` / `[key=seikatsu-hogo]` には `taxableIncomeMin`/`Max` が**存在しない**。課税所得だけで `brackets` を線形探索すると、非課税世帯（上限24,600円）が `kazei-under380`（44,400円）に落ちる。入力9（世帯の課税状況）を先に見る。→ T-21
4. **★2026-08-01 の分岐★**（`amendments[0].impact`）。`beforeAug2026` / `fromAug2026` のどちらを既定にするかは**公開日ではなく試算対象月**で決める。`Date.now()` を直接参照せず注入可能な時計にする。→ T-25 / T-26
5. **★多床室（室料を徴収しない老健等）の第3段階②だけ据置★**（`fromAug2026.note`）。「第3段階②は居住費 +100円」と一般化すると 430 → 530 と誤る。`kyojuhi[].stage3b` を**必ず配列から引く**。→ T-28
6. **★超過分に負担割合を掛けない★**（`kubunShikyuGendo.description`「これを超えた分は全額自己負担」）。→ T-15
7. **★超過分と食費・居住費を高額介護サービス費の上限計算に入れない★**（`kougakuKaigoServiceHi.note`）。→ T-24
8. **`kyojuhi[].stage1: 0` を falsy 判定しない**。多床室（特養等）の第1段階は 0円（無料）であり、未設定ではない。`??` ではなく `in` / `undefined` チェックを使う。→ T-29
9. **`unit` フィールドの意味が区分ごとに違う**: `"世帯"` / `"個人"` / `"世帯/個人 併記"`。`hikazei-80man` だけ `limit`（世帯 24,600）と `limitIndividual`（個人 15,000）の**両方**を持つ。片方だけ表示しない。→ T-22
10. **`kubunShikyuGendo.levels` に `keikateki`（経過的要介護）が含まれる**。要介護度セレクトを `yoshien1`〜`yokaigo5` の7件で決め打ちすると落ちる。`levels[]` からループで生成する。→ T-14
11. **`yokaigoNintei.levels` と `kubunShikyuGendo.levels` の key が完全一致しない**。前者には `higaitou`（非該当）が含まれ、後者には `keikateki` が含まれる。**join するときの欠落を扱う**（非該当は限度額がなく、経過的要介護は基準時間の定義がない）。
12. **`futanWariai.brackets` の順序に依存しない**。JSON は 3割 → 2割 → 1割 の順だが、`rate` で `find` する。1割の bracket は全しきい値が `null`（フォールバック）。
13. **年金収入は「収入」、その他は「所得」**（入力7 のラベル）。混同すると桁が変わる。UI で明確に分ける。
14. **補足給付の「世帯」には世帯分離した配偶者を含む／年金収入に非課税年金（遺族年金・障害年金）を含む**（`userStages.note`）。入力17 のヒントに必ず書く。**通常の税制上の世帯・年金収入と定義が違う。**
15. **`verify` / `verify.expect` はデータ検証用**。ツールの表示ロジックで参照しない。
16. **`boundaryDiscrepancy.value` は `"未確認"` という文字列**。数値としてパースしない。

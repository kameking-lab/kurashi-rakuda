## 1. 概要

### 1.1 何を解くツールか

「今日なに作ろう」を**考えない**で済ませる。条件（人数・調理時間・除外したい食材・和洋中）を入れると、**1週間分の献立（主菜・副菜・汁物）**が即座に1画面に出る。気に入らなければ引き直す。日単位でも引き直せる。

解いているのはレシピ検索ではない。**意思決定の回数を7回から1回に減らすこと**が価値である。

### 1.2 競合との差別化

[02_ツール一覧](../../docs/02_ツール一覧.md) の記述に基づく差別化根拠のみを記す（憶測で競合情報・需要を書かない）。

| 出典箇所 | 記述 | 本ツールでの意味 |
|---|---|---|
| #40 行 ツール名 | 「献立自動提案（条件指定・1週間分・**静的データ＋クライアント抽選**）」 | アーキテクチャそのものが仕様。サーバー側の推薦API・ログイン・DBを持たない |
| #40 行 備考 | 「**Web完結ツールほぼ不在**の空白。**アプリDL不要が差別化**」 | 「まず会員登録」「まずアプリを入れて」を挟まない。URL を開いた次の瞬間に献立が出る |
| 優先ツール TOP10 #10 | 「献立自動提案（#40）— Web 完結ツール不在。**『考えなくていい』価値の象徴**」 | ブランド体現ツール。UI が「考えさせる」設計になった時点で失敗 |
| 棲み分け方針 2. ② | 「スケジュール・段取り・チェックリスト系（非計算）」 | 本ツールはこの系統。金額計算・制度判定を持ち込まない |

### 1.3 このツールが `data/seido/` を使わない理由

- YMYL **低**（[02_ツール一覧](../../docs/02_ツール一覧.md) #40 の YMYL 列）。制度・お金・医療の判断を含まない。
- したがって `data/seido/` 配下のデータセット・`seido-common.schema.json` の `sourceId` / `checkedAt` / `verify` の枠組みは**適用しない**。
- ただし **出典・免責の思想は流用する**: レシピデータにも `source`（自作／出典）を持たせ、`asOf` を持ち、免責文を必須にする（§8）。
- **栄養価・健康効果・アレルギー安全性の判断は一切しない**（§2.2）。ここを踏み越えた瞬間に YMYL 高のツールになり、`data/seido` と同等の出典管理が必要になる。

## 2. スコープ

### 2.1 含むもの

- 1週間（7日）分の献立の自動生成（1日 = 主菜1・副菜1・汁物1）
- 条件指定: 人数 / 1日の合計調理時間の上限 / 除外食材 / ジャンル / 汁物の要否
- **決定的な抽選**（同じ条件＋同じシードなら必ず同じ献立。§4.2）
- 引き直し（週全体 / 1日だけ / 1品だけ）
- 主菜の7日間重複回避、調理法・主材料の連日重複の回避（§4.5）
- 食材の使い回し提示（同じ食材を数日内で使い切る組み合わせを優先。§4.6）
- URL への条件＋シードの保存（共有・ブックマークで同じ献立が再現できる）
- 買い物リストへの拡張余地（#44。§9.6）、冷凍保存期間との部品共通化（#37。§9.7）

### 2.2 含まないもの（明示）

| 含まないもの | 理由 |
|---|---|
| **栄養価の計算・表示（カロリー/PFC/塩分/野菜g）** | 断定できる根拠を持たないため。表示すれば「この献立は栄養バランスが良い」という主張になる。★[03_コンテンツ戦略](../../docs/03_コンテンツ戦略.md) §4.1 の「医療判断はスコープ外」方針と同じ線引き★ |
| **健康効果の主張**（「疲労回復に」「産後の回復に」等） | 同上。タグにも入れない（§3.4 の禁止タグ） |
| **アレルギー対応・アレルゲン判定** | 「卵不使用」と表示した瞬間に安全性の保証になる。除外は**食材IDによる機械的な除外**に留め、§8 の免責を必須にする |
| 妊娠中・授乳中・離乳食の可否判定 | #12（妊娠中のNG食品）・#19（離乳食の量）の担当範囲。本ツールからリンクのみ |
| レシピの手順文・分量の詳細 | 本ツールは**献立を決める**ツール。レシピ本文は外部（#38 調味料換算・#42 味付け黄金比・#43 人数スケール）に委ねる |
| ユーザーのレシピ投稿・お気に入り保存（サーバー側） | 「静的データ＋クライアント抽選」（#40）。保存は localStorage のみ（#52 相当、Phase 2 の P2-06） |
| 冷蔵庫の残り物からの提案 | #49「食材使い切りナビ」の担当範囲 |
| 買い物リストの生成 | #44（優先度A）の担当範囲。本ツールは**データ構造だけ先に用意する**（§9.6） |
| 献立の履歴・「先週と被らない」 | Phase 2（P2-06 localStorage）。MVP は毎回独立 |
| 朝食・昼食・弁当 | MVP は**夕食のみ**。#51 お弁当箱容量は別ツール |

## 3. 静的データのスキーマ設計

配置: `data/kondate/`（★`data/seido/` ではない★ §1.3）

```
data/kondate/
├── recipes.json        # レシピ本体
├── ingredients.json    # 食材マスタ
└── meta.json           # 版・免責・件数
```

### 3.1 `ingredients.json` — 食材マスタ

食材名を**文字列で持たない**。ID で正規化する。「玉ねぎ」「たまねぎ」「タマネギ」が別物になると除外条件が破綻するため。

```json
{
  "version": "1.0.0",
  "asOf": "2026-07-17",
  "ingredients": [
    {
      "id": "onion",
      "name": "玉ねぎ",
      "aliases": ["たまねぎ", "タマネギ", "オニオン"],
      "category": "vegetable",
      "shelf": "room",
      "excludable": true,
      "commonness": 5
    },
    {
      "id": "egg",
      "name": "卵",
      "aliases": ["たまご", "タマゴ", "鶏卵"],
      "category": "egg",
      "shelf": "fridge",
      "excludable": true,
      "commonness": 5
    },
    {
      "id": "soy-sauce",
      "name": "しょうゆ",
      "aliases": ["醤油"],
      "category": "seasoning",
      "shelf": "room",
      "excludable": false,
      "commonness": 5
    }
  ]
}
```

| フィールド | 型 | 必須 | 説明 |
|---|---|---|---|
| `id` | string `^[a-z0-9-]+$` | ✔ | 一意。レシピから参照する唯一のキー |
| `name` | string | ✔ | 表示名（1つに固定） |
| `aliases` | string[] | | 検索・入力補完用。**除外判定は id で行い、aliases は入力を id に解決する用途のみ** |
| `category` | enum | ✔ | `meat` / `fish` / `vegetable` / `mushroom` / `seaweed` / `soy` / `egg` / `dairy` / `grain` / `noodle` / `seasoning` / `other` |
| `shelf` | enum | ✔ | `room` / `fridge` / `freezer`。★#37 冷凍保存期間との共通化キー（§9.7）★ |
| `excludable` | boolean | ✔ | 除外条件に指定できるか。`seasoning` は原則 `false`（しょうゆを除外すると候補が消滅する） |
| `commonness` | integer 1–5 | ✔ | 常備度。5=常備前提（塩・しょうゆ）。買い物リスト（#44）で「買わなくていい」判定に使う |

### 3.2 `recipes.json` — レシピ本体

```json
{
  "version": "1.0.0",
  "asOf": "2026-07-17",
  "recipes": [
    {
      "id": "buta-shogayaki",
      "name": "豚のしょうが焼き",
      "course": "main",
      "cookTimeMin": 15,
      "genre": "japanese",
      "method": "fry",
      "mainIngredientId": "pork-loin",
      "mainCategory": "meat",
      "ingredients": [
        { "id": "pork-loin", "role": "main" },
        { "id": "onion", "role": "sub" },
        { "id": "cabbage", "role": "garnish" },
        { "id": "ginger", "role": "sub" },
        { "id": "soy-sauce", "role": "seasoning" }
      ],
      "tags": ["kids-ok", "one-pan", "make-ahead-ng"],
      "servingsBase": 2,
      "scalable": true,
      "season": ["all"],
      "source": "original",
      "weight": 3
    }
  ]
}
```

| フィールド | 型 | 必須 | 説明 |
|---|---|---|---|
| `id` | string `^[a-z0-9-]+$` | ✔ | 一意 |
| `name` | string | ✔ | 表示名 |
| `course` | enum: `main` / `side` / `soup` | ✔ | ★献立の骨格★ 1日 = main×1 + side×1 + soup×1 |
| `cookTimeMin` | integer 1–120 | ✔ | 目安の調理時間（分）。**下ごしらえ含む、炊飯・つけ置きは除く** |
| `genre` | enum: `japanese` / `western` / `chinese` / `other` | ✔ | ジャンル条件の絞り込みに使う |
| `method` | enum: `grill` / `fry` / `simmer` / `steam` / `deepfry` / `boil` / `dress` / `raw` | ✔ | 調理法。**連日重複の回避**に使う（§4.5）。★コンロ・鍋の同時使用の推定にも使う（§4.7）★ |
| `mainIngredientId` | string | ✔ | 主材料の食材ID。`ingredients` に含まれること |
| `mainCategory` | enum | ✔ | `mainIngredientId` の食材の `category` と一致すること（冗長だが抽選が高速になる。CI で整合を検査。§9.3） |
| `ingredients[]` | object[] | ✔ | 使う食材。★分量は持たない（§2.2）★ |
| `ingredients[].id` | string | ✔ | `ingredients.json` の id |
| `ingredients[].role` | enum: `main` / `sub` / `garnish` / `seasoning` | ✔ | 使い回し判定は `main` / `sub` のみで行う（しょうゆの共有を「使い回し」と呼ばない） |
| `tags` | string[] | | §3.4 の許可リストのみ |
| `servingsBase` | integer | ✔ | 基準人数（通常2） |
| `scalable` | boolean | ✔ | 人数スケール可否。`false` の場合は人数条件で「2人分のまま」と注記 |
| `season` | enum[]: `spring` / `summer` / `autumn` / `winter` / `all` | ✔ | 旬。**MVP では抽選に使わない**（§4.9） |
| `source` | enum: `original` / string | ✔ | `original` = 自作。外部由来なら出典を明記 |
| `weight` | integer 1–5 | ✔ | 抽選の重み。5=定番、1=変化球。**決定的抽選のために整数**（§4.3） |

### 3.3 `meta.json`

```json
{
  "version": "1.0.0",
  "asOf": "2026-07-17",
  "counts": { "main": 0, "side": 0, "soup": 0 },
  "disclaimer": "…（§8）",
  "algorithmVersion": 1
}
```

★`algorithmVersion` はシードの再現性の境界を表す★ アルゴリズムまたは `recipes.json` の内容が変わると、同じシードでも同じ献立にならない。URL には `v`（= `version` + `algorithmVersion`）を含め、不一致なら「レシピが更新されたため、同じ献立を再現できません」と表示する（§4.10）。

### 3.4 タグの許可リスト（★これ以外を書いたら CI で落とす★）

| タグ | 意味 |
|---|---|
| `kids-ok` | 子どもが食べやすい味付け（辛くない）。★「子どもに良い」ではない★ |
| `one-pan` | フライパン／鍋ひとつ |
| `no-fire` | 火を使わない（レンジ・和えるのみ） |
| `make-ahead-ok` / `make-ahead-ng` | 作り置きの向き不向き。★#45 作り置き日持ち一覧との共通化キー（§9.7）★ |
| `freezer-ok` | 冷凍できる。★#37 冷凍保存期間との共通化キー★ |
| `budget` | 材料費が抑えめ |
| `quick` | `cookTimeMin <= 10`（CI で自動整合を検査） |

**禁止タグ（CI の禁止語 lint に入れる）**: `healthy` / `low-calorie` / `diet` / `balanced` / `nutritious` / `immune` / `for-pregnancy` / `for-baby` / `allergy-free` / `egg-free` / `gluten-free` ほか、栄養・健康・アレルギーを主張する一切の語（§2.2）。

### 3.5 データ量の下限（★抽選が成立する最小件数★）

| course | 最小件数 | 根拠 |
|---|---|---|
| `main` | **7 以上**（絶対条件） | 主菜の7日間重複回避（§4.5 H2）が原理的に不可能になる |
| `main` | **30 以上**（推奨） | ジャンル条件（4種）× 調理時間条件で絞られても7品残るため。§4.8 の候補枯渇を実用上回避 |
| `side` | 20 以上 | 連日の `method` 回避のため |
| `soup` | 10 以上 | 同上 |

MVP のリリース基準は **main 30 / side 20 / soup 10 = 60件**。CI でこの下限を検査する（§9.3）。

## 4. 抽選アルゴリズム（決定的記述）

### 4.1 入出力

```
generate(conditions, seed, data) -> WeekMenu | Failure
```

```
WeekMenu = { days: [ Day × 7 ], seed, version, warnings[] }
Day      = { index: 0..6, main: Recipe, side: Recipe, soup: Recipe | null, totalTimeMin }
Failure  = { reason: "no-candidate", course, dayIndex, relaxHint }
```

**`generate` は純関数である。** `Math.random()`・`Date.now()`・`localStorage`・DOM を参照してはならない（§9.1）。

### 4.2 乱数シードの扱い（★再現性★）

| 決定事項 | 内容 |
|---|---|
| PRNG | **mulberry32**（32bit、実装10行、決定的、依存ゼロ） |
| シードの型 | `uint32` |
| シードの出所 | ①URL の `?s=` パラメータ ②無ければ生成時に `crypto.getRandomValues` で1回だけ取得し、**即座に URL へ書き戻す**（`history.replaceState`） |
| 再現性の保証 | `(conditions, seed, version, algorithmVersion)` が同一なら**出力は必ず同一**。ブラウザ・OS・実行時刻に依存しない |
| 「引き直し」 | **シードを変えるだけ**。`seed' = mulberry32(seed)()` を新シードとし、URL を更新する。→ 戻るボタンで前の献立に戻れる |
| 1日だけ引き直し | `dayIndex` を混ぜた派生シード `seed_day = hash32(seed, dayIndex, nonce)` でその日だけ再抽選。`nonce` は引き直し回数。**他の6日は絶対に動かさない**（§4.11） |
| 1品だけ引き直し | `hash32(seed, dayIndex, courseIndex, nonce)`。同日の他2品を固定する |

★なぜ URL に書くのか★ 「アプリDL不要が差別化」（#40）。サーバー保存もログインも無いのに、**URL をコピーすれば夫に送れる／ブックマークすれば来週も同じ献立が出る**。これが「Web完結」の実装上の意味である。

### 4.3 重み付き抽選（決定的）

`weight` を使った重み付け抽選は、**候補の並び順に依存してはならない**（並び順が変われば結果が変わり、`recipes.json` の行の入れ替えだけで再現性が壊れる）。

```
1. 候補集合を id の辞書順で安定ソートする       ← ★必須★ 入力順に依存させない
2. 累積重み配列 C を作る（C[i] = Σ weight[0..i]）
3. r = floor( rng() * C[last] )                  ← rng() ∈ [0,1)
4. 二分探索で C[i] > r を満たす最小の i を返す
```

### 4.4 全体フロー

```
STEP 0  条件を正規化（食材名 → id に解決。aliases も引く）
STEP 1  course ごとに「条件を満たす候補プール」を作る（§4.5 の H1・H3・H4）
STEP 2  各プールが空 → Failure("no-candidate") を即返す（§4.8）
STEP 3  main を7日分、重複なしで選ぶ（§4.5 H2）
STEP 4  日ごとに side・soup を選ぶ（§4.5 S1〜S3 のスコアで）
STEP 5  1日の合計時間の上限（§4.5 H4）を満たすか検査。満たさなければ STEP 4 を再試行（上限あり。§4.8）
STEP 6  食材使い回しのスコアで並べ替え（§4.6）
STEP 7  警告を集めて返す
```

**すべてのループは決定的な順序（day 0→6、course は main→side→soup）で回す。** 並列化・非同期化しない。

### 4.5 制約: ハード（絶対）とソフト（優先）

#### ハード制約 — 1つでも破ったら出力してはならない

| ID | 制約 | 実装 |
|---|---|---|
| **H1** | **除外食材を含むレシピは絶対に出さない** | `recipe.ingredients[].id` のいずれかが `excludeIds` に含まれたら候補から**除去**（減点ではない）。★`role` を問わない。`seasoning` でも `garnish` でも除外する★ |
| **H2** | **主菜は7日間で重複しない** | 選択済み `main.id` を集合で保持し、プールから除く |
| **H3** | ジャンル条件に合致 | `genre` 条件が `any` 以外なら `recipe.genre == 条件`。★`soup` は例外的にジャンル不問とする（味噌汁は洋食の日にも出る）。この例外を明示的に書く★ |
| **H4** | 1日の合計調理時間 ≦ 上限 | `main.cookTimeMin + side + soup ≦ maxTotalTimeMin`（§4.7 の注意） |
| **H5** | `soup` 不要なら汁物を出さない | `soup = null`。H4 の合計から除く |

#### ソフト制約 — スコアで優先。満たせなくても出力する

| ID | 制約 | スコア |
|---|---|---|
| **S1** | 前日と同じ `method` を避ける | 同じなら **−30** |
| **S2** | 前日と同じ `mainCategory` を避ける（肉が3日続かない） | 同じなら **−20**、2日前とも同じなら追加 **−20** |
| **S3** | 同じ日の main / side / soup で `method` が重ならない | 重複1組につき **−10**（コンロが足りない） |
| **S4** | 食材の使い回し（§4.6） | 共有食材1つにつき **+15**（上限 +45） |
| **S5** | `weight` の高いレシピを優先 | `weight × 5` |

**スコアの合計が同点の場合**: `id` の辞書順で小さい方を採る（★決定性のため必須★）。

### 4.6 食材の使い回し（S4）

「木曜に半分残ったキャベツを土曜に使い切る」を作る。

```
S4スコア(candidate, dayIndex) =
    15 × | { i ∈ candidate.ingredients | i.role ∈ {main, sub} }
          ∩ { 前後2日以内に選ばれた献立の main/sub 食材 } |   （最大3、上限+45）
```

- `role = seasoning` / `garnish` は**数えない**（しょうゆの共有は使い回しではない）。
- `commonness = 5`（常備前提）の食材も**数えない**（塩を共有しても買い物は減らない）。
- ★`+15` を大きくしすぎると全部同じ食材の献立になる。S1（−30）を超えないこと。上限 +45 は S1+S2（−50）を超えないよう設計している。★
- UI では「キャベツを木・土で使い切れます」と**理由を明示する**。これが #44 買い物リストへの布石になる（§9.6）。

### 4.7 合計調理時間（H4）の注意

`main + side + soup` の**単純合計**とする。

- ★実際は並行調理するので合計より短くなる。だが「並行できる時間」を推定すると根拠のない数値になる。★
- したがって UI のラベルは「合計 45分」ではなく「**目安 45分（順番に作った場合）**」とする。
- S3（同日の `method` 重複を減点）は、この「並行しやすさ」をスコアで間接的に扱う唯一の手段である。数値には反映しない。

### 4.8 候補が0件のときの挙動（★必ず定義する★）

| 状況 | 挙動 |
|---|---|
| STEP 2 でプールが空（例: 「10分以内 × 中華 × 肉を除外」） | **Failure("no-candidate")**。空の献立・部分的な献立を**返さない** |
| STEP 3 で main が7日分揃わない（プールが7未満） | Failure。「主菜の候補が **N品** しかありません（7日分には7品必要です）」と**具体的な件数を出す** |
| STEP 5 の再試行が上限（**20回**）に達した | Failure("time-limit")。「1日の合計時間を **N分**にすると作れます」と、**実際に候補から計算した最小可能値**を提示する |
| ソフト制約を全部破っても出せる | **出す**（ソフトは出力可否に影響しない）。ただし `warnings` に「主菜が2日続けて肉になっています」等を入れる |

**`relaxHint` は具体的に**: 「条件を緩めてください」ではなく「**調理時間を20分→30分にすると、候補が3品→18品になります**」。これを出すために、Failure 時に**各条件を1つずつ外した場合の候補数を再計算する**（`main` プールのみ。7回程度の再計算で済む）。

### 4.9 MVP で使わないフィールド

| フィールド | 理由 |
|---|---|
| `season` | 抽選に使わない。「今は夏だから」の判定は体感とずれやすく、地域差もある。**データには持つが、MVP のロジックは参照しない**。Phase 2 で「旬の食材を優先」トグルとして追加候補 |
| `servingsBase` / `scalable` | 人数条件は**表示のみ**に使う（「4人分に換算すると材料は2倍です」）。抽選には影響しない。分量計算は #43 レシピ人数スケール換算へリンク |

### 4.10 バージョン不一致（★再現性の限界★）

- URL には `?s=<seed>&v=<version>.<algorithmVersion>&c=<conditions>` を持つ。
- `v` が現在のデータと不一致 → **献立を再現できたふりをしない**。「レシピが更新されたため、この URL の献立は再現できません。同じ条件で新しく作りますか？」と明示する。
- ★これを黙って新しい献立を出す実装にすると、「夫に送った URL と自分の画面が違う」が起きる。★

### 4.11 引き直しの独立性（★UX の核心★）

| 操作 | 動くもの | 動かないもの |
|---|---|---|
| 週まるごと引き直し | 7日すべて | — |
| 1日だけ引き直し | その日の main / side / soup | **他の6日は1品も動かない** |
| 1品だけ引き直し | その1品 | **同じ日の他2品を含め、他はすべて動かない** |

- 1日だけの引き直しでも **H1（除外食材）・H2（主菜の重複回避）は維持する**。他の6日の main を「使用済み」として除外プールに入れること。
- ★実装の罠★ 「1日だけ引き直したら、別の日と主菜が被った」は**バグ**。TC-08 で検証する。

## 7. エラー・未確認データの扱い

### 7.1 エラーの分類

| 種別 | 例 | 挙動 |
|---|---|---|
| **バリデーションエラー** | `servings = 0`、`excludeIds` 11件、`excludable: false` の食材の除外 | 入力欄でインライン指摘。抽選を実行しない |

| **時間制約の詰まり**（`time-limit`） | 20回再試行しても H4 を満たせない | 「合計時間を N分にすると作れます」の具体値を返す |

### 7.2 「未確認データ」に相当するもの

制度データを使わないため `data/seido` の未確認管理は適用されない（§1.3）。本ツールで「わからないこと」は次のとおりで、いずれも**書かないことで扱う**。

| わからないこと | 扱い |
|---|---|
| そのレシピが栄養的に良いか | **書かない**（§2.2）。タグにも入れない（§3.4 の禁止タグ） |
| そのレシピがアレルギー的に安全か | **書かない**。§8 の免責で原材料確認を求める |
| 実際の調理時間 | 「**目安**」と書く。並行調理の短縮は推定しない（§4.7） |
| 材料費 | `budget` タグ（相対的な傾向）のみ。**金額を書かない** |
| 旬 | データには持つが抽選に使わない（§4.9） |

### 7.3 レシピデータの品質ルール（データ整備者向け）

[queue/hoikuryo-backlog.md](../../queue/hoikuryo-backlog.md) の「絶対ルール」の思想を流用する（対象は制度データではないが、規律は同じ）。

- **推測で埋めない。** `cookTimeMin` を「たぶん15分」で入れない。実際に作れる手順から見積もる。
- **`source` を必ず書く。** `original`（自作）でないなら出典を明記する。
- **`method` / `mainCategory` を目分量で付けない。** 抽選のソフト制約が直接これに依存する（S1・S2）。
- 新しい食材を足したら `ingredients.json` に**先に**追加する（TC-34 が落ちる）。

## 9. 実装メモ（罠・データ更新時に壊れる箇所）

### 9.1 `generate` を純関数に閉じ込める

```
packages/calc/kondate/
├── generate.ts      # ★純関数。import は型と PRNG のみ★
├── prng.ts          # mulberry32
├── score.ts         # S1〜S5
└── generate.test.ts # §6
```

- `generate.ts` から `Math.random` / `Date` / `window` / `localStorage` / `fetch` を**参照しない**。データは引数で受け取る。
- ★これを守らないと §6.2 の全テストが成立しない。TC-13 で機械的に検査する。★
- シードの生成（`crypto.getRandomValues`）と URL 操作は**呼び出し側（UI 層）**の責務。

### 9.2 決定性を壊す3つの罠

| 罠 | 症状 | 対策 |
|---|---|---|
| **候補配列を入力順のまま使う** | `recipes.json` に1件足しただけで、既存の全 URL の献立が変わる | §4.3 の「id 辞書順に安定ソート」。TC-12 |
| **`Object.keys` / `Map` の列挙順に依存** | 環境・ビルドで結果が変わる | 列挙順に依存する箇所を作らない。同点は id 辞書順（§4.5）。TC-17 |
| **`Array.prototype.sort` の不安定性** | 同じスコアの並びが処理系依存 | 比較関数で必ず id を第2キーにする（安定ソートに頼らない） |

### 9.3 CI で検査すること（データ追加のたびに走る）

| 検査 | テスト |
|---|---|
| 参照整合（`ingredients[].id` / `mainIngredientId`） | TC-33, TC-34 |
| `mainCategory` と食材の `category` の一致 | TC-33 |
| タグの許可リスト・禁止タグ | TC-35 |
| `quick` タグと `cookTimeMin` の整合 | TC-36 |
| `course` 別の件数下限（30/20/10） | TC-37 |
| UI 文言の禁止語（カロリー・栄養バランス・アレルギー対応・◯◯不使用） | TC-38 |

[08_タスク分解](../../docs/08_タスク分解.md) P1-05「CI パイプライン（テスト・Lighthouse CI・忠実性ゲート・sponsored lint）」に、上記の禁止語 lint を相乗りさせる。

### 9.4 データ更新時に壊れる箇所

| いつ | 壊れる箇所 |
|---|---|
| **レシピを1件でも追加・削除・`weight` 変更** | ★過去に発行した全 URL の献立が変わる★ → `meta.json` の `version` を上げ、§4.10 のバージョン不一致画面を出す。**黙って変えない** |
| `algorithmVersion` を上げる（スコア調整含む） | 同上。S1〜S5 の重みを1つ変えただけでも `algorithmVersion` を上げる |
| `ingredients.json` から食材 id を削除 | 古い URL の `excludeIds` が未知 id になる → TC-24（無視して警告）で吸収済み |
| `course` の件数が下限を割る | TC-37 で CI が落ちる。**リリースを止める** |
| `method` の enum に値を追加 | S1・S3 のスコア分布が変わる → `algorithmVersion` を上げる |

### 9.5 パフォーマンス

- 60件（MVP）〜300件（Phase 2 想定）程度なら、STEP 3〜5 の全探索でもクライアントで数ms。**最適化しない。**
- ★`recipes.json` は静的アセットとして配信し、初回ロードでメモリに載せる。抽選ごとの fetch はしない。★ これが「アプリDL不要」（#40）の体感速度の正体。
- 引き直しは**再計算のみ**（ネットワークアクセスゼロ）。オフラインでも引き直せる。

### 9.6 #44 買い物リスト自動生成への拡張余地

[02_ツール一覧](../../docs/02_ツール一覧.md) #44「買い物リスト自動生成（献立連動）」は優先度 **A**（Phase 2）、備考「**#40 の拡張**」。本ツールは #44 を**作らないが、作れる形で作る**。

| 本仕様の要素 | #44 での使われ方 |
|---|---|
| `WeekMenu.days[].{main,side,soup}.ingredients[]` | 7日分を集約 → 食材IDごとに重複排除するだけでリストになる |
| `ingredients[].role` | `seasoning` を除外して「買うもの」だけにする |
| `ingredients.commonness` | **5（常備前提）を除外**して「買い足すもの」にする。★このフィールドは #44 のためだけに今 持っておく★ |
| `ingredients.category` | 売り場順（野菜→肉→魚→…）の並べ替え |
| `ingredients.shelf` | 「冷蔵」「冷凍」の保管先グルーピング |
| §4.6 の使い回し情報 | 「キャベツ1玉（木・土で使用）」の**まとめ買い提示** |

★**分量は持たない**（§3.2）ため、#44 は「何を買うか」までしか出せない。「何g」を出すには `ingredients[].amount` の追加が必要で、それは `servingsBase` × `scalable` × #43（レシピ人数スケール換算）と連動する別設計になる。**MVP でスキーマに `amount` を足す誘惑に負けないこと**（60件全部に正確な分量を入れる作業が発生し、S級ツールがB級の作業量で沈む）。★

### 9.7 #37 冷凍保存期間・#45 作り置き日持ちとの部品共通化

[02_ツール一覧](../../docs/02_ツール一覧.md) #37「冷凍保存期間検索（食材150種）」は優先度 **S**（同じ MVP 20本）、備考「プログラマティック展開の主力」。#45「作り置き 日持ち一覧」は備考「#37 と部品共通」。

| 共通部品 | 共有のしかた |
|---|---|
| **食材マスタ** | `data/kondate/ingredients.json` の `id` / `name` / `aliases` / `category` を **#37 と共有する**。★#37 は150種、本ツールの MVP は60レシピ分（数十種）。**#37 の食材マスタを上位集合とし、本ツールがその部分集合を参照する**設計にする★ |
| `shelf`（`room`/`fridge`/`freezer`） | #37 の保存場所軸と同じ enum を使う |

## 1. 概要

### 1.1 何を解くツールか

「自分の住む自治体で、認可保育所に子どもを預けたら**月いくら**払うのか」を、自治体・所得・年齢区分・保育必要量・きょうだい構成から求める。

本ツールの回答は 3 パターンに分かれ、**どれも正常系**として扱う。

| パターン | 例（収集済み12自治体） | 出す答え |
|---|---|---|
| ① 階層表から金額が出る | 横浜・大阪(〜R8/8)・札幌・福岡・川崎・名古屋 | 月額の目安金額＋該当階層＋根拠 |
| ② 無償のため 0 円 | 東京23区6区（世田谷・練馬・大田・江戸川・足立・杉並）、大阪(R8/9〜) | 「保育料は0円です」＋**給食費・延長保育料の案内**（§1.2） |
| ③ データ未収集・未確認 | 上記12自治体以外の全自治体、足立区のD階層境界 | 金額を出さず、確認先を案内（§7） |

### 1.2 競合との差別化（[02_ツール一覧](../../docs/02_ツール一覧.md) の記述の範囲で）

- 同 #21 の記述: 「**全国対応ツールの空白地帯**。自治体データが参入障壁=堀になる」。優先ツール TOP10 では 2 位「空白地帯。データ整備が参入障壁になる本命」。
- 同 §棲み分け方針: 本サイトは「女性のライフイベント文脈型」を作る。保育料は kurashi-keisan.jp の既存160ツールと重複しない領域（送客リストにも載っていない）。
- 従って差別化の実体は **UIの巧拙ではなくデータの網羅と正確さ**。以下を仕様の柱にする。
  1. 全ての金額に**その自治体の公式資料**（`sources[]`）へのリンクを併記する（[08_タスク分解](../../docs/08_タスク分解.md) P1-04「出典3点セット」）。
  2. **無償の自治体で「0円」と言い切れること**自体が価値。競合の記事型コンテンツは階層表がある前提で古い情報が残りやすい。0円と判明した自治体では給食費・延長保育料へ話題を移す（§1.1 パターン②）。
  3. **未確認を未確認と表示する**（§7）。推測値を出さない。

※ 需要・競合の詳細は [02_ツール一覧](../../docs/02_ツール一覧.md) の記述以上のことを本仕様書に書かない。

## 2. スコープ

### 2.1 含むもの

- 収集済み自治体（`data/seido/hoikuryo/` 配下、2026-07-17時点で **12自治体**）の**認可保育所**の月額保育料（利用者負担額）の試算。
- 年齢区分 `under3` / `age3plus` × 保育必要量 `standard`（保育標準時間）/ `short`（保育短時間）の 4 象限。
- 階層判定（`tiers[].incomeMin` / `incomeMax` / `isWelfare` / `isNonTaxable`）。
- 税年度の切替（`bracketBasis.taxYearRule`）に基づく「どの年度の課税額を入れるべきか」の案内。
- 無償化（`freeTuition`）の反映と、多子軽減・ひとり親等軽減の**存在の案内**（§2.2 参照。金額の自動適用はしない）。
- 対象外自治体・未確認項目の明示（§7）。

### 2.2 含まないもの（明示）

| 含まないもの | 理由・根拠 |
|---|---|
| 給食費（主食費・副食費）、延長保育料、教材費・行事費、私立園の上乗せ徴収 | 全12ファイルの `scope.excludes` に一貫して除外と明記。自治体・施設ごとに異なり階層表の外側 |
| 小規模保育・家庭的保育・事業所内保育など施設種別ごとの料金差 | backlog §2.4。現スキーマに `facilityType` 軸がない。川崎は原典に3系統の料金列があるが認可保育所列のみ収録（`kanagawa-kawasaki.json` の `bracketBasis.note`）。MVP は認可保育所のみ |
| 認可外保育施設・認証保育所・幼稚園（1号）の利用料・補助 | 全ファイルの `scope.excludes` |
| **多子軽減後の確定金額の自動計算** | backlog §2.6 と各ファイルの `freeTuition.multiChildPolicy`。自治体ごとに減額方式が全く違う（横浜=第2子専用の金額列／川崎=半額/名古屋=1/2かつ階層でカウント方法が変わる／札幌・福岡・大阪=第2子以降0円）。スキーマの `tiers[].fees` は**第1子（基準額）のみ**を保持している。→ **第2子以降は金額を断定せず、`multiChildPolicy` の原典テキストを提示する**（例外は §4 ステップ7 の「第2子以降0円」が `secondChild.value === 0` で明示されている自治体） |
| ひとり親世帯等の軽減後の金額 | backlog §2.6。スキーマに対応フィールドがなく、原典表記が `bracketBasis.note` に文章で保存されているだけ。→ 該当の可能性を告知し、原典テキストと窓口を案内 |
| 0歳児と1・2歳児の金額差 | backlog §2.5。現スキーマの `under3` は統合区分。杉並は令和7年度表で0歳と1・2歳が別額だったが現在双方0円のため欠落なし（`tokyo-suginami.json` の `bracketBasis.note`）。**有料かつ差がある自治体を収集する前にスキーマ分割を判断すること**（§9） |
| 保育料の減免制度（失業・疾病・災害等） | 札幌・川崎とも「各区への個別相談事項」として数値が公表されておらず、両ファイルの `localExtension` に「未確認」と記録 |
| 収集済み12自治体以外の全自治体 | 未収集。backlog §1 のキューに従い順次追加 |
| 入園可否・保活の選考（利用調整指数） | 別ツール（[02_ツール一覧](../../docs/02_ツール一覧.md) #23 保活スケジュールメーカー）。**練馬区の利用調整用の所得割ルールを保育料に転用しないこと**（backlog §4） |

## 4. 計算式（公式資料との対応付け）

記法: `M` = 選択した自治体の JSON（`data/seido/hoikuryo/<id>.json`）。

### ステップ0: 年度・データ鮮度の検査

```
if (M.fiscalYear !== 試算対象月の年度) → §7 の「年度が違うデータ」警告を最上部に固定表示
if (M.amendments[].status === "under-review") → §7 の「未確定」警告を表示
```
- 根拠: `M.fiscalYear`、`M.amendments[]` ／ 出典: 各ファイルの `sources[]`
- 具体例（名古屋）: `aichi-nagoya.json` の `fiscalYear: 2025`、`amendments[0].summary`「令和8年度版が未公表(2026-07-17確認)」／出典: 「令和7年度利用者負担額（保育料）基準月額表（2・3号認定子ども（3歳未満児））」（名古屋市子ども青少年局）。**令和8年度の試算には使えない**（同 `impact`）。
- 具体例（福岡）: `fukuoka-fukuoka.json` の `amendments[0].summary`「表題に『（案）』が付されたまま」／出典: 「令和8年度 保育料表（案）（福岡市）」。

### ステップ1: 施行日による分岐（無償化の予定）

```
for a of M.amendments:
  if (a.status === "scheduled" && 試算対象月 >= a.effectiveFrom && a が全額無償を定める) → 保育料 = 0 円で確定し、以降のステップを行わない
```
- 根拠: `osaka-osaka.json` の `amendments[0]`（`effectiveFrom: "2026-09-01"`, `impact`「令和8年9月分以降は全階層・全年齢区分の保育料が0円になる。ツールで令和8年9月以降の月を試算する場合は tiers を参照せず一律0円を返すこと」）／出典: 「令和8年度 保育料のお知らせ（私立保育所）」（大阪市こども青少年局）
- **大阪市の `tiers` が有効なのは令和8年4月分〜8月分のみ**（同 `impact`、`scope.includes[0]`）。

### ステップ2: 年齢区分による分岐

```
if (入力の年齢区分 === "age3plus" && M.freeTuition.age3plusFree.value === true) → 保育料 = 0 円
```
- 根拠: `M.freeTuition.age3plusFree.value` ／ 出典: `M.sources[]` のうち `M.freeTuition.age3plusFree.sourceId` が指すもの（例 横浜: 「令和8年度 横浜市保育所等利用料のご案内（令和8年4月版）」）
- 収集済み12自治体すべてで `true`。ただし**ハードコードせず必ず値を読む**。

### ステップ3: 課税状況による階層判定（★所得割額より先★）

★backlog §2.2★ A/B/C はいずれも所得割額0円のため、金額では判別できない。次の順で決める。

```
switch (入力5 の課税状況):
  welfare      → M.tiers.find(t => t.isWelfare === true)
  nonTaxable   → M.tiers.find(t => t.isNonTaxable === true)
  equalOnly    → M.tiers.find(t => !t.isWelfare && !t.isNonTaxable && t.incomeMin === 0 && (t.incomeMax === 1 || t.incomeMax === 48600 相当の最下位)) ※下記注
  incomeTaxed  → ステップ4 へ
```
- 根拠: `M.tiers[].isWelfare` / `M.tiers[].isNonTaxable`
- **注（`incomeMax` の表現が自治体間で不統一）**: backlog §2.2 の通り、この不統一はスキーマで吸収しない。ツールが吸収する。実装は「均等割のみ課税世帯に対応する階層」を次の優先順で解決する。
  1. `bracketBasis.note` / `tiers[].label` に「均等割のみ」「所得割非課税」の記載がある階層（横浜 `C`／川崎 `C1`／福岡 `C1`／名古屋 `Ｃ1階層`／大阪 `第3`／足立 `C階層`）
  2. 札幌は**例外**: `hokkaido-sapporo.json` の `bracketBasis.note`「パンフレット※2 に『所得割が非課税で、均等割のみ課税されている世帯はＣ階層に該当します』とあるため、均等割のみ課税世帯（所得割0円）は B ではなく C1（所得割48,600円未満）に該当する」→ `C1`（`incomeMin: 0, incomeMax: 48600`）
  3. 該当階層を機械的に特定できない場合は金額を出さず §7 へ
- 具体値（横浜 `C`）: `under3.standard = 6,700` / `short = 6,500`／出典: 「令和8年度 横浜市保育所等利用料金表」


## 9. 実装メモ（罠・端数処理・データ更新で壊れる箇所）

### 9.1 端数処理

- **保育料に端数処理は不要。** `tiers[].fees.*` は自治体が公表した確定額（円単位の整数）をそのまま持つ。四捨五入・切り捨てを一切かけない。
- **前処理（6/8 換算等）の端数は本ツールで確定できない。** 大阪・川崎・札幌・福岡・江戸川の 6/8 換算について、端数処理の方法は各 `bracketBasis.note` に記載がない。**★2026-07-17 修正★** 当初は「ツールが換算して換算後の額を表示する」設計としていたが、換算率を保持する構造化フィールドがスキーマに無く（散文の `note` のみ）、端数規則も原典に無い。→ **ツールは換算を代行しない。** 上級者モードの入力欄のラベルを「前処理を済ませた所得割額」とし、同じ画面に `bracketBasis.note` / `deductionsIgnored` / `taxYearRule` を提示してユーザーが換算する（T-10 / T-28）。推測で切り捨て/四捨五入を実装しない。`bracketBasis` に換算率と端数規則が構造化されれば、自動換算と「境界±数円で両隣の階層を併記」を実装できる（そのときまで保留）。
- 川崎の第2子（半額）は `82,800 × 0.5 = 41,400` のように割り切れるが、割り切れない階層の端数規則は原典に記載がない。→ §2.2 の通り**自動計算しない**。

### 9.2 罠（優先度順）

1. **★所得割額の直接入力★**（backlog §2.1）。既定は年収推計、上級者モードのみ直接入力。前処理の説明を同画面に必ず出す。→ T-10 / T-28
2. **★A/B/C を金額で判別しない★**（backlog §2.2）。`isWelfare` / `isNonTaxable` と入力5を先に見る。→ T-06〜T-08 / T-29
3. **★札幌の C1 例外★**。「均等割のみ課税 → B（0円）」と一般化すると 11,000円 を 0円と誤る。→ T-16
4. **★階層表がある前提で書かない★**（backlog §2.3）。練馬の `tiers.length === 1`、東京23区の全階層0円。→ T-23 / T-27
5. **★大阪の 2026-09-01★**。`tiers` を参照してはいけない期間がある。→ T-14
6. **★名古屋の年度★**。`fiscalYear: 2025`。バナーを折りたたみ可能にしない。→ T-22
7. **★川崎の定額減税だけ向きが逆★**。適用**後**の額で算定（他自治体は適用**前**）。共通ロジックに畳まない。
8. **境界の +1 規約の二重適用**。横浜のみ原典「〜以下」→ +1 済み。他自治体は原典「未満」そのまま。ツール側で再度 +1 すると横浜が全階層1つずれる。→ T-01〜T-04 / T-11
9. **最下位 D 階層の `incomeMin: 0`**（江戸川 D1・杉並 D1・足立 D階層）。`tiers` を上から線形探索して最初に一致した階層を返す実装だと、A/B/C を飛ばして D に落ちる。ステップ3 を必ず先に通す。
10. **`incomeMax: null` と `incomeMax: 0` の取り違え**。札幌・川崎・福岡の A/B は `incomeMax: 0`（＝どの値も `income < 0` にならず絶対にマッチしない）。これは「フラグで判定せよ」という意図の表現であり、バグではない。`incomeMax: 0` を `null` と誤読しない。
11. **`fees` は `under3` / `age3plus` とも optional**（スキーマ上 `required` は `tier` / `incomeMin` / `incomeMax` / `fees` のみ）。`fees.under3?.standard` の undefined を必ず扱う。
12. **`freeTuition.multiChildPolicy.secondChild.value` の型が混在**（number 0 ／ number 0.5 ／ 長文 string）。`typeof` で分岐する。`countingRule.value` も string / null（練馬）が混在。
13. **`tiers[].tier` は自治体表記のまま**（`"Ａ階層"` は**全角Ａ**、`"第8(8A)"`、`"C1"`）。tier 名で機械分岐しない。全角/半角の正規化も勝手にしない（表示は原文）。
14. **`municipalityCode`** は6桁チェックデジット込み。ルーティングキーに使うなら `id`（`tokyo-setagaya` 等）と混同しない。

### 9.4 検算者への指示（実装者 ≠ 検算者）

[08_タスク分解](../../docs/08_タスク分解.md)「YMYL の金額計算（S/A）は必ず『実装者≠検算者』のダブルチェックを構造化する」に従い、検算者は次を独立に行う。

1. §6 の全テストケースの期待値を、実装コードを見ずに `data/seido/hoikuryo/*.json` から直接引いて再計算する。
2. §5 の境界表の全行を、対応する JSON の `incomeMin`/`incomeMax`/`fees` と1対1で突き合わせる。
3. §4 の各ステップの「根拠: <JSONパス>／出典: <タイトル>」が実在することを確認する（JSONパスの typo、出典タイトルの不一致を探す）。
4. **仕様書中にハードコードされた数値が `data/seido/` に実在しない場合は、仕様書のバグとして差し戻す。**

# /tools 探索性の情報設計（docs/13 A-5/A-6/B-8 対応方針）

作成: 2026-07-20 司令塔（フェイブル）。実装はオーパス次波（DIAG-A3）。
制約（ミッション指定）: SSR で全件描画を維持／既存 URL 不変／SEO 非破壊／広告臭ゼロ。

## 0. 決定

**「/tools 一覧の強化」と「カテゴリページ新設」の両方を採用する。**
二者択一にしない理由: 一覧強化は「探す人」（サイト内回遊）の問題を、カテゴリページは「着地する人」（検索流入・共有・パンくず遡行の 404 = docs/13 B-8）の問題を解くもので、解決する課題が別。カテゴリページは URL の「追加」であり既存 URL は不変（非破壊）。

## 1. /tools 一覧の強化（探索性）

### 1.1 検索
- 一覧最上部に検索ボックスを置く（トップのヒーロー検索と同じ照合ロジック: title/description/solves の部分一致）。
- **挙動はクライアント側の絞り込み**: SSR では全 64 件を今までどおり全件描画し、入力に応じて JS で非該当カードを `hidden` にする（DOM から消さない）。JS 無効時は全件表示のまま = SSR 全件・SEO 非破壊。
- **Enter キーの仕様（トップ・一覧共通）**: Enter で `/tools?q=<入力値>` へ遷移する。/tools は `q` があれば初期状態でその絞り込みを適用（サーバーでは何も変えず、ハイドレーション後に適用。適用中は「◯件が一致」を aria-live で通知）。これで docs/13 A-5（Enter 無反応 =「壊れている」体験）を解消し、絞り込み結果 URL が共有可能になる。
- 0 件時: 「見つかりませんでした。カテゴリから探す／全ツールを見る」を表示（煽り文言なし）。

### 1.2 カテゴリ・ライフステージの固定ナビ
- 現行のカテゴリ見出しアンカーとライフステージ chips を、スクロール追従の 1 行バー（sticky, top: ヘッダー高）に統合する。モバイルは横スクロール chips。
- docs/13 B-9 のスティッキーヘッダー透け問題と同時に解消（背景を不透明トークンに）。

### 1.3 パーソナライズの一覧反映
- **SSR の並び順は変えない**（検索エンジン・JS 無効・CLS の安定性を優先）。
- 反映は 2 段: ①一覧最上部に「あなた向け」枠（トップと同じ推薦ロジック・最大 6 件）をクライアント側で描画 ②各カードに「あなた向け」バッジをクライアント側で付与。既存 `relatedOnly`（関係あるものだけ表示）トグルを一覧にも適用する。
- 並び替えそのものを行わないことで、パーソナライズ設定者と非設定者で DOM 順序が変わらず、ハイドレーション競合（docs/13 A-3）の再発面を増やさない。

## 2. カテゴリページ新設（/tools/[category]/page.tsx）

- 7 カテゴリの SSG ページを新設。内容: カテゴリ見出し＋一言（既存トークンの文体）／当該カテゴリの全ツールカード（SSR 全件）／関連ガイド記事（同カテゴリの記事から新しい順に 6 本）／他カテゴリへの導線。
- メタ: title「◯◯の無料ツール一覧 | くらしのラクダ」・canonical・BreadcrumbList JSON-LD。sitemap に 7 URL を追加。
- 既存の `/tools#category` アンカー導線は当面残し、ヘッダー/フッター/パンくずのカテゴリリンクは新 URL へ向ける（アンカー URL への流入も引き続き機能する = 非破壊）。
- /guide/[category] も同型で新設する（記事側の 404 も同根のため。実装は同一 PR でなくてよい）。

## 3. 実装分解（オーパス向け）

1. IA-1: /tools 検索ボックス＋ `?q=` 対応＋ Enter 遷移（トップの検索フォームに form action="/tools" を付与）
2. IA-2: sticky カテゴリ/ライフステージバー＋ヘッダー不透明化
3. IA-3: パーソナライズ「あなた向け」枠とバッジの /tools 適用（relatedOnly 含む）
4. IA-4: /tools/[category] SSG 7 ページ＋sitemap＋内部リンク切替
5. IA-5: /guide/[category] 同型（IA-4 のテンプレ流用）

完了条件（共通）: SSR HTML に全件が残ること（JS 無効で全ツール到達可能）／既存 URL が全て 200 のまま／Lighthouse Perf・A11y 90 以上維持／check:brand PASS／Playwright で「q= 共有 URL で絞り込み済み表示」「Enter 遷移」「sticky バー」「あなた向け枠」の 4 点を実機確認。

## 4. やらないこと

- SSR での並び替えパーソナライズ（SEO・CLS・キャッシュ共有と相性が悪い）
- 検索の外部サービス・検索結果の専用ページ（現規模ではクライアント絞り込みで足りる。64→200 本規模になったら再設計）
- 人気順・ランキング表示（brand-lint の禁止語„ランキング“と同思想。「おすすめ」枠は既存の「まず使ってほしい」文体を維持）

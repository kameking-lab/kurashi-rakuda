# 05_技術構成 — くらしのラクダ（仮称）

作成日: 2026-07-17 ／ 前提: 実装は Phase 1 以降。本書は構成の確定のみ。

---

## 1. 設計原則

1. **ランニングコスト最小**: API 課金は原則ゼロ。無料枠・既存契約（Vercel Pro、ConoHa VPS）の範囲で運用。追加固定費は**ドメイン代のみ**を目標
2. **既存アセット最大活用**: kurashi-keisan / anzen-ai-portal で実証済みのパターンをそのまま移植。新技術の検証はしない
3. **速さ＝ブランド**: 広告なし・計測タグ最小により、構造的に競合より速いサイトを作る（LCP 2.0s 以内、CLS ほぼゼロ）
4. **ツールはすべてクライアント計算**: 計算ロジックは TypeScript 純関数としてブラウザ内で完結。サーバーコスト・レイテンシ・個人情報リスクを同時に消す

## 2. 技術スタック

| レイヤ | 採用 | 理由 |
|-------|------|------|
| フレームワーク | Next.js（App Router、SSG 中心） | kurashi-keisan と同一。知見・コンポーネント再利用 |
| ホスティング | Vercel Pro（既存契約） | 追加費用ゼロ。ISR でツール改修を即時反映 |
| 言語 | TypeScript strict | 計算ロジックの正確性担保（ユニットテスト必須） |
| スタイリング | Tailwind CSS | 既存サイトと共通。デザイントークンは本サイト専用に定義（[04_サイト設計](04_サイト設計.md)） |
| コンテンツ | Markdown/MDX をリポジトリ内管理（Git=CMS） | CMS 費用ゼロ。anzen-ai 記事工場の出力先をそのまま接続 |
| DB | 原則なし（Phase 1〜2）。必要時は Vercel Postgres/KV 無料枠 | ツールはステートレス。保存系機能（後述）は localStorage 優先 |
| 検索 | Pagefind（静的検索、ビルド時生成） | サーバー不要・無料でサイト内検索を提供 |
| 分析 | GA4 + Google Search Console（anzen-ai の設定パターン流用） | 無料。計測タグはこれのみ（速度と信頼のため最小構成） |
| 監視/自律改善 | ConoHa VPS 上の自律改善エンジン（kurashi-keisan のものを横展開） | Lighthouse 定点観測、リンク切れ検知、GSC クエリ→ツール候補抽出 |
| Push/通知 | Web Push（anzen-ai 実装流用）。Phase 2 以降で予防接種リマインダー等に使用 | 無料。アプリ不要で「アプリ級」の便利さを出す |

## 3. AI・API 課金の扱い（原則ゼロ）

- **実行時 API 課金ゼロ**: サイト上でユーザーのリクエストごとに LLM API を叩く機能は Phase 1〜2 では作らない。AI は「制作時」（記事工場・平易文変換・データ整備）にのみ使用し、既存パイプラインの範囲で賄う
- 週数連動 ToDo・献立提案などの「賢く見える機能」は、**事前生成した静的データ＋クライアントロジック**で実装する（例: 妊娠40週×週ごと ToDo は 40 レコードの静的 JSON）
- 将来 RAG 相談機能（kakomon-ai 転用）を載せる場合も、Phase 3 で費用対効果を再審査してから。本書時点では非採用

## 4. リポジトリ構成（モノレポ、実装フェーズで作成）

```
kurashi-rakuda/
├── docs/                 # 本企画ドキュメント群
├── apps/web/             # Next.js 本体
│   ├── app/(tools)/[category]/[slug]/   # ツールページ（プログラマティックSEO）
│   ├── app/(articles)/
│   └── components/tools/ # 入力フォーム・結果表示の共通UI
├── packages/calc/        # 計算ロジック純関数＋テスト（フレームワーク非依存）
├── packages/data/        # 制度データ・静的JSON（改定年度をメタデータに持つ）
└── ops/                  # VPS 自律改善エンジン設定・記事工場接続
```

- `packages/calc` を独立させるのは kurashi-keisan での学び。ロジックをテスト可能・移植可能に保ち、将来の相互流用（kurashi-keisan ⇄ 本サイト）を容易にする
- `packages/data` の制度データは「改定年度」「出典URL」「次回改定チェック日」をフロントマターに必須で持つ（E-E-A-T 対応。[03_コンテンツ戦略](03_コンテンツ戦略.md)）

## 5. プログラマティック SEO の実装方式

- kurashi-keisan で実証済みの「1テンプレート×パラメータ展開」方式を採用
- 例: 保育料計算 → 自治体別ページを政令市・中核市から段階展開／予防接種スケジュール → 生年月日別ではなく「月齢別」静的ページ＋クライアント側パーソナライズ
- 展開はビルド時 SSG。Vercel Pro のビルド上限に収まるよう、1展開軸あたり上限を設けて段階リリース（詳細は [08_タスク分解](08_タスク分解.md)）

## 6. 既存サイトとの相互送客設計

| 起点 | 導線 | 実装 |
|------|------|------|
| kurashi-keisan の女性向けツール | ページ下部「もっと詳しい○○は くらしのラクダへ」 | 関連ツールカード（共通JSONで管理し双方向に自動生成） |
| 本サイトの汎用計算ニーズ | 「単位換算などの汎用計算は kurashi-keisan へ」 | 同上（逆向き） |
| anzen-ai-portal | 防災・安全×家庭のクロス記事から相互リンク | 記事フロントマターの related_sites キー |
| instagram/note 自動運用 | ツールのスクショ＋1行ハウツーを自動投稿 → プロフィールリンク | Phase 3。既存 PDCA エンジンに本サイトのアカウントを追加 |

- 相互送客は「関連があるときだけ」表示。無関係な自社サイトバナーの常設は広告臭になるため**しない**

## 7. コスト見積（月額）

- Vercel Pro: 既存契約内（増分ゼロ想定。ビルド分数のみ監視）
- ConoHa VPS: 既存契約内（ジョブ追加のみ）
- ドメイン: .jp 新規取得＋更新のみが純増コスト（実勢価格は [04_サイト設計](04_サイト設計.md) の調査結果を参照）
- API 課金: 0円（原則）。記事工場の生成コストは anzen-ai 側の既存運用に含まれる範囲で
- **月額純増: ドメイン代のみ（年額数千円）**

## 8. 品質ゲート（実装フェーズで CI に組み込む）

- 計算ロジック: `packages/calc` はテストカバレッジ 100% を必須（YMYL の金額計算を含むため）
- 制度データ: 「次回改定チェック日」超過で CI が警告 → 自律改善エンジンが Issue 起票
- パフォーマンス: Lighthouse CI で Performance/Accessibility 90 未満はマージ不可
- 記事: anzen-ai の CI 忠実性ゲート（出典照合）を通過したもののみデプロイ
- 紹介リンク: `rel="sponsored"` と開示文の存在を lint で機械チェック（[06_紹介ポリシー](06_紹介ポリシー.md) の技術的担保）

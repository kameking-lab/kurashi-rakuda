## 2. 恒久策の比較

### 案A: Vercel Git 接続（推奨）
- 内容: Vercel ダッシュボード → Project Settings → Git → `kameking-lab/kurashi-rakuda` を接続。Production Branch = master
- 効果: master への push（=PR マージ）ごとに自動で本番デプロイ。PR にはプレビュー URL が自動発行され検収も楽になる。`VERCEL_GIT_COMMIT_SHA` が環境変数で入るため、本番がどのコミットかをページ側に埋め込める（§4）
- コスト: **社長作業 約2分**（ダッシュボードでの接続操作のみ。リポジトリ Public のため権限は GitHub OAuth で完結）
- リスク: master への push が即公開になる（現行運用は「CI green の PR しかマージしない」ため実質リスクなし。緊急停止はダッシュボードの Instant Rollback で可能）

### 案B: GitHub Actions からの自動デプロイ
- 内容: `vercel deploy --prod --token=$VERCEL_TOKEN` を CI の build 成功後ステップに追加
- 効果: 「全ゲート green のときだけデプロイ」を Actions 側で強制できる
- コスト: 社長が Vercel トークンを発行し GitHub Secrets（VERCEL_TOKEN/VERCEL_ORG_ID/VERCEL_PROJECT_ID）を登録（約10分）。トークンの失効管理が恒久的に必要
- リスク: シークレット管理という新しい運用面。Public リポジトリでの token 扱いは fork PR に流出しない設計確認が必要（pull_request では secrets 非公開だが設定ミスの余地）

### 案C: 運用ルールのみ（現行の暫定）
- 内容: 司令塔が検収サイクル毎（最低 24 時間に 1 回）`npx vercel --prod --yes` を実行し、§4 の実測照合まで行う
- 効果: 追加設定ゼロで今日から回る（実際に 2026-07-20 から運用中）
- リスク: **今回の事故の原因そのもの**（人・セッション依存）。司令塔不在の日に再発する

## 3. 決定と推奨

- **推奨: 案A（Vercel Git 接続）**。理由: 社長作業 2 分で人依存を根絶でき、プレビュー URL で検収も強化される。案B は案A で不足が出た場合（例: verify-seido --fetch まで通らないと公開したくない等）の将来オプション
- **案A が有効になるまでは案C を厳守**（司令塔の検収サイクル終端で必ず本番反映 + 実測照合。報告に「Vercel本番反映済み」を明記）
- 社長作業（2分）: Vercel ダッシュボード → kurashi-rakuda → Settings → Git → Connect Git Repository → GitHub → `kameking-lab/kurashi-rakuda` → Production Branch を `master` に設定

## 4. ドリフトの機械検知（接続方式によらず実施）

1. **本番 SHA の埋め込み**: `app/layout.tsx` に `<meta name="build-sha" content={process.env.VERCEL_GIT_COMMIT_SHA ?? process.env.BUILD_SHA ?? "local"} />` を追加（案A なら自動で入る。CLI デプロイ時は `BUILD_SHA=$(git rev-parse HEAD) npx vercel --prod` 相当で渡す）→ 実装はオーパス次波（DIAG-S1 の子タスク）
2. **日次照合**: 既存の verify-seido.yml（日次 schedule）に「本番の build-sha meta を curl し、origin/master の HEAD と不一致なら Issue 起票」のジョブを追加 → 同上
3. それまでの暫定照合: 本番の保育料自治体数（SSG HTML の実測）と `data/seido/hoikuryo/*.json` 件数の一致確認を検収サイクル毎に手動実施（2026-07-20 復旧時の実測手順を踏襲)

## 5. 検収サイクルの標準手順（司令塔）

1. `git checkout master && git pull` → 検収対象 PR を全ゲートで検収（CI green を自己申告と扱わず、ローカルで `npm run check` + fidelity + verify-seido 構造 + フレッシュビルドを再実行）
2. マージ後の master で `npx vercel --prod --yes`
3. 本番実測: 保育料自治体数の一致（暫定）または build-sha 一致（§4 実装後）
4. 報告に「Vercel本番反映済み（本番実測: ○○）」を明記

# 15. マスコット画像の権利方針（docs/13 B-6 / DIAG-B6。決裁は社長）

作成: 2026-07-20 司令塔（フェイブル）。事実関係は診断（docs/13 §1 B-6・プライバシー監査）に基づく。

## 1. 事実関係

- マスコット「らっく」10 ポーズ（public/mascot/*.png）とヒーロー画像は、**Gemini の消費者向け Web UI をブラウザ自動操作（CDP）で生成**したもの（mascot/browser-generation.log に経緯記録。API 課金は不使用）。
- プロンプトは mascot/prompts*.txt に全記録。生成物の管理台帳（配置規約・重複・解像度）は scripts/mascot-manifest.mjs + CI check:mascot で機械管理済み。
- リポジトリは Public のため、生成経緯ログは誰でも読める。ログ 1 行目にローカル Windows ユーザー名を含むパスが露出している（軽微・docs/13 C 級）。
- リポジトリ内に画像の権利帰属・ライセンスを記録するファイルは無い。サイト上に AI 生成の旨の表記も無い（現行日本法上、表記義務は無い）。

## 2. リスク評価

1. **利用規約リスク（主）**: 消費者向け Gemini Web UI の自動アクセスは Google の利用規約の自動化禁止条項と抵触するおそれがある。生成物の権利自体は Google は主張しない建付けだが、「規約違反の手段で得た素材」という攻撃面が Public リポジトリで自己開示されている状態。
2. **著作権リスク（従）**: AI 生成画像は日本法上、人間の創作的寄与が乏しければ著作権が発生しない可能性があり、第三者の模倣を止めにくい（ブランド保護の弱さ）。他者著作物との類似リスクはキャラクターデザインが独自であるため低いと評価。
3. **信頼リスク**: 「誠実さを売る」ブランドが生成経緯を伏せていると受け取られる余地。逆に、明示すれば差別化要素にもなる。

## 3. 選択肢

### 案A: 現状維持＋記録整備（コスト最小）
- mascot/ASSETS.md（仮）に由来・生成日・プロンプト参照・利用方針を明文化し、/about に「イラストは AI 生成をもとに調整しています」の一文を追加。ログのローカルパス行は削除（履歴には残る旨を注記）。
- 残るもの: 規約リスク（過去の生成手段は変えられない）。

### 案B: API 課金経由で再生成（推奨・中コスト）
- 商用利用条件が明確な画像生成 API（Gemini API 等の有償利用）で同プロンプト系から 10 ポーズを再生成し、置き換える。プロンプト資産が既にあるため作業は機械的。デザインの微差はマニフェスト検収（check:mascot）と目視で吸収。
- 効果: 規約リスクを将来分について解消（「現行画像は正規 API で生成」と言える状態にする）。費用は API 従量で小さい。
- 残るもの: 旧画像の生成経緯は履歴に残る（隠さない方針とセットで問題を最小化）。

### 案C: 人間イラストレーターへ発注（コスト大）
- 著作権の確実な帰属（契約で譲渡）とブランド保護の最大化。費用・納期・今後のポーズ追加の都度コストが発生。
- 「完全無料・儲け先行させない」運営で先行投資に見合うかは規模次第。

## 4. 推奨

**短期: 案A を即実施（次波ソネットで可能）。中期: 案B を実施し、案A の記録に「正規 API で再生成済み」を追記。案C は独自ドメイン取得・ブランドが軌道に乗った段階の再検討事項。**
理由: 実害の中心は「規約グレーな生成経緯が Public で自己開示されている」ことであり、記録整備＋正規 API 再生成で攻撃面をほぼ閉じられる。デザイン資産（プロンプト）は失われない。

## 5. 社長決裁事項

1. 案A/B/C の選択（推奨: A 即時 + B 中期）
2. /about への AI 生成明示の文言可否
3. 案B の場合: 使用する API と課金アカウントの用意
